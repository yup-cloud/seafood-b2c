# 수령/출고 운영 설계

## 1. 문서 목적

이 문서는 손님이 상품을 받는 방식을 `직접 픽업`, `퀵`, `택배`로 나누어 운영할 때 필요한 화면, 상태, 예외 처리 기준을 정리한 문서다.

핵심 목표는 세 가지다.

- 주문 접수 단계에서 필요한 정보를 빠짐없이 받는다
- 포장 완료 이후 어떤 주문을 어떻게 넘겨야 하는지 헷갈리지 않게 한다
- 고객 문의가 왔을 때 현재 수령 상태를 바로 답할 수 있게 한다

## 2. 수령 방식 정의

### 2-1. 직접 픽업

- 고객이 매장에 직접 방문해서 받아가는 방식
- 핵심 정보:
  - 방문 예정 시간
  - 주문자명
  - 연락처

### 2-2. 퀵

- 고객 또는 매장이 퀵 기사로 당일 전달받는 방식
- 핵심 정보:
  - 수령인명
  - 수령인 연락처
  - 주소
  - 퀵 요청 시간 또는 수령 희망 시간
  - 퀵비 부담 주체 메모

### 2-3. 택배

- 당일 또는 익일 택배로 발송하는 방식
- 핵심 정보:
  - 수령인명
  - 수령인 연락처
  - 주소
  - 우편번호
  - 냉장/냉동/아이스포장 메모
  - 송장번호

### 2-4. 택배 세부 유형

- 일반택배: 평일 상시 가능
- 당일택배: 평일 오전 주문 마감이 있는 방식
- 고속택배: 터미널 또는 고속 연계 전달 방식

## 3. 수령 방식별 권장 가공 규칙

### 직접 픽업

- 원물, 오로시, 회 모두 가능
- 회 수령도 가장 자연스러운 방식

### 퀵

- 회, 오로시, 원물 모두 가능
- 회 주문은 퀵이 가장 권장되는 수령 방식
- 기사 호출 시간과 인계 가능 시간을 함께 받아야 함

### 일반택배

- 오로시/필렛 중심 권장
- 회 상태 발송은 강한 경고 또는 제한이 필요
- 진공포장 선택이 권장됨

### 당일택배

- 오로시/필렛 위주 권장
- 회 발송은 품목과 거리 기준으로 운영자 승인 필요

### 고속택배

- 회보다는 오로시/필렛 위주 권장
- 도착 후 수령 방식 안내가 필요

## 4. 고객 주문서 설계 기준

### 직접 픽업 선택 시

- 방문 예정 시간 입력
- 주소 입력은 숨김 또는 비필수 처리

### 퀵 선택 시

- 수령인 정보와 주소 입력 필수
- 퀵 요청 메모 입력 가능
- 당일 배차 가능 시간 안내 필요

### 택배 선택 시

- 수령인 정보와 주소 입력 필수
- 우편번호 입력
- 발송 후 송장 안내 문구 표시
- 신선식품 특성상 발송 가능 요일/시간 안내 필요

## 5. 관리자 운영 흐름

### 4-1. 공통 흐름

1. 주문 접수
2. 입금 확인
3. 손질 진행
4. 포장 완료
5. 수령 방식별 인계 또는 발송 처리

### 4-2. 직접 픽업 흐름

1. 포장 완료 후 `직접 픽업 대기`
2. 고객 방문
3. 인수 확인 후 `픽업 완료`

### 4-3. 퀵 흐름

1. 포장 완료 후 `퀵 출고 대기`
2. 주소와 연락처 최종 확인
3. 퀵 기사 호출 또는 고객 측 퀵 배정 확인
4. 기사 인계 후 `퀵 인계 완료`

### 4-4. 택배 흐름

1. 포장 완료 후 `택배 발송 대기`
2. 송장 생성 또는 접수
3. 송장번호 입력
4. 발송 후 `택배 발송 완료`

## 6. 권장 상태 구조

### 주문 상태

- 신규 접수
- 입금 대기
- 입금 검토 필요
- 손질 대기
- 손질 중
- 포장 완료
- 인계/출고 대기
- 완료
- 취소

### 수령/출고 상태

- 직접 픽업 대기
- 직접 픽업 완료
- 퀵 출고 대기
- 퀵 인계 완료
- 택배 발송 대기
- 택배 발송 완료

## 7. 마감 시간과 운영 정책

### 퀵

- 픽업 2~3시간 전 주문 또는 확정 필요

### 일반택배

- 평일 위주 운영
- 주말 운영 여부를 별도 설정할 수 있어야 함

### 당일택배

- 오전 마감 시간 설정 필요
- 도착 시간은 확정값이 아니라 안내 문구로 처리해야 함

### 품목별 사전 주문

- 특정 품목은 전일 예약 또는 당일 오전 마감이 필요할 수 있음

## 8. 운임/포장 정책

- 운임은 수령 방식과 중량에 따라 달라질 수 있음
- 10kg 이상 추가 요금 같은 규칙을 메모 또는 정책으로 관리
- 포장비 포함/별도 여부를 명확히 표시
- 진공포장 가능 여부를 선택값으로 둔다

## 9. 화면 요구사항

### 6-1. 주문 상세 화면

- 수령 방식
- 수령인명
- 수령인 연락처
- 주소
- 퀵 메모
- 택배 송장번호
- 출고 상태

### 6-2. 출고 관리 화면

- 직접 픽업 / 퀵 / 택배 탭 분리
- 포장 완료 시간 기준 정렬
- 빠른 완료 처리 버튼
- 주소/연락처 즉시 확인 가능

### 6-3. 모바일 운영 포인트

- 퀵 기사 연락 시 바로 주소 확인 가능
- 택배 송장번호를 휴대폰에서도 입력 가능
- 픽업 완료 버튼을 한 번에 누를 수 있어야 함

## 10. 예외 상황 처리

### 7-1. 고객이 수령 방식을 바꾸는 경우

- 포장 전이면 변경 허용
- 포장 후 변경이면 관리자 확인 필요

### 7-2. 퀵 주소가 불명확한 경우

- 출고 대기 상태 유지
- 연락 필요 메모 남김

### 7-3. 택배 발송이 당일 불가한 경우

- 다음 발송 예정일 메모
- 고객 안내 필요 상태 표시

### 7-4. 직접 픽업 지연

- 픽업 대기 시간이 길어진 주문 표시
- 냉장/냉동 보관 메모 필요

### 10-5. 택배와 가공 방식이 맞지 않는 경우

- 일반택배 + 회 조합은 경고 또는 관리자 승인 필요
- 고객에게 오로시/필렛 전환 제안을 할 수 있어야 함

- address_detail
- dispatch_note
- tracking_number
- handed_off_at

## 12. 구현 우선순위

### 1단계

- 주문서에서 수령 방식 선택
- 주문 상세에 수령 정보 저장
- 수령 방식별 필수 입력 분기
- 가공 방식 경고 문구 반영

### 2단계

- 출고 상태 분리
- 출고 관리 화면 구현
- 택배 세부 유형과 마감 규칙 반영

### 3단계

- 송장번호 기록
- 픽업 지연 / 출고 지연 표시
- 진공포장/운임비 정책 연결

## 13. 완료 기준

- 직접 픽업, 퀵, 택배 주문이 섞여도 누락 없이 처리된다
- 포장 완료 이후 남은 주문을 한 화면에서 볼 수 있다
- 고객 문의 시 현재 수령 상태를 바로 답할 수 있다
- 일반택배에 부적합한 주문 조합을 사전에 경고할 수 있다
