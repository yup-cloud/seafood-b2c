# 자동 입금확인 기능 상세 설계

## 1. 기능 목적

주문별 기대 입금 정보와 실제 계좌 입금 내역을 자동으로 대조해서, 사장님이 계좌 앱과 주문 목록을 반복해서 번갈아 보지 않도록 만드는 기능이다.

핵심 목표는 아래 두 가지다.

- 입금 확인에 쓰는 반복 시간을 줄인다
- 잘못된 자동 확정은 줄이고, 사람이 봐야 하는 주문만 남긴다

## 2. 기본 원칙

- 자동화는 `전부 자동 승인`보다 `자동 대조 + 검토 필요 분리`가 우선이다
- 입금이 불명확한 건은 반드시 사람이 마지막 확인을 할 수 있어야 한다
- 누가 언제 어떤 기준으로 입금 확인을 했는지 기록이 남아야 한다
- 최종 금액이 아직 확정되지 않은 주문은 자동 입금확인 대상이 아니다
- 미입금 주문은 출고 완료 처리 전에 다시 한 번 경고해야 한다

## 3. 권장 도입 방식

### 3-1. 1단계: 반자동 대조

- 계좌 입금 내역을 웹앱으로 불러온다
- 주문별 금액, 입금자명, 주문 생성 시간과 비교한다
- 정확히 일치하는 건은 `자동 확인 완료`
- 애매한 건은 `검토 필요`

### 3-2. 2단계: 실시간 또는 주기적 자동 동기화

- 은행 또는 핀테크 연동을 통해 입금 내역을 자동으로 가져온다
- 새 입금 건이 들어오면 주문과 자동 대조한다

### 3-3. 3단계: 고도화

- 가상계좌 또는 고유 입금 식별 규칙 적용
- 반복 고객 패턴 학습
- 동일 금액 다중 주문 구분 정확도 개선

## 4. 입금 상태 설계

### 주문 기준 상태

- 최종 금액 확정 전
- 미입금
- 자동 확인 완료
- 검토 필요
- 수동 확인 완료
- 부분 입금
- 초과 입금
- 환불 필요

### 거래 기준 상태

- 미매칭 거래
- 단일 주문 매칭
- 다중 후보 충돌
- 수동 처리 완료

## 5. 자동 대조에 필요한 데이터

### 주문 데이터

- 주문번호
- 주문자명
- 입금자명
- 기대 입금 금액
- 최종 금액 확정 시간
- 주문 생성 시간
- 입금 마감 시간

### 거래 데이터

- 거래 발생 시간
- 입금 금액
- 입금자명
- 계좌 정보
- 거래 고유 ID

## 6. 매칭 로직 초안

### 6-1. 1순위 일치 조건

- 금액이 정확히 일치
- 입금자명이 정확히 일치
- 입금 시간이 주문 생성 이후이며 허용 시간 범위 내

이 조건을 모두 만족하면 `자동 확인 완료` 후보로 본다.

### 6-2. 2순위 보조 조건

- 주문자명과 입금자명이 유사
- 같은 연락처로 들어온 최근 주문 존재
- 반절 매칭 주문처럼 두 건이 함께 확정된 묶음 주문 여부

이 조건은 점수 계산에 반영하되, 자동 확정이 아니라 `검토 필요`로 보내는 기준에 가깝다.

### 6-3. 자동 확정 금지 조건

- 같은 금액의 미입금 주문이 여러 건 존재
- 입금자명이 비어 있거나 의미 없는 값인 경우
- 부분 입금 의심
- 초과 입금 의심
- 주문 금액이 아직 최종 확정되지 않은 경우

## 7. 반절 매칭 주문과의 연동

반절 매칭 주문은 일반 주문보다 자동 입금확인 규칙을 더 엄격하게 잡는 것이 안전하다.

권장 방식:

- 두 반절 주문이 모두 `매칭 완료`가 된 뒤에만 입금 요청
- 각 주문은 개별 금액으로 입금 받음
- 한쪽만 입금한 경우 매칭 그룹 단위로 `검토 필요` 처리
- 두 주문 모두 입금이 맞아야 작업 단계로 이동

## 8. 관리자 화면 설계

### 8-1. 대시보드 카드

- 미입금 건수
- 자동 확인 완료 건수
- 검토 필요 건수
- 부분 입금 건수

### 8-2. 입금 검토 큐

관리자가 바로 확인해야 하는 주문만 모아보는 화면이다.

표시 항목:

- 주문번호
- 주문자명
- 입금자명
- 기대 금액
- 실제 입금 금액
- 입금 시각
- 추천 매칭 점수
- 검토 사유

### 8-3. 주문 상세 화면

- 기대 입금 정보
- 최종 금액 확정 메모
- 실제 연결된 거래 정보
- 자동 대조 결과
- 수동 승인 버튼
- 수동 반려 버튼
- 예외 메모
- 출고 차단 또는 경고 표시

## 9. 예외 상황 처리

### 9-1. 입금자명 불일치

- 금액이 같아도 바로 자동 확정하지 않음
- 검토 필요로 보내고 주문자명, 연락처, 주문 시간까지 함께 보여줌

### 9-2. 같은 금액 주문이 여러 건인 경우

- 자동 확정 금지
- 후보 주문 목록을 함께 보여주고 사람이 선택

### 9-3. 부분 입금

- 상태를 `부분 입금`으로 표시
- 잔액 안내 또는 관리자 메모 필요

### 9-4. 초과 입금

- 상태를 `초과 입금`으로 표시
- 환불 또는 차액 처리 기록 필요

### 9-5. 중복 입금

- 같은 주문에 두 건 이상이 연결되면 자동 승인 금지
- 환불 또는 추가 주문 여부 확인 필요

## 10. 현실적인 구현 방식 비교

### 방식 A. 거래 내역 수동 업로드

- 장점: 빠르게 시작 가능
- 단점: 완전 자동은 아님

적합 상황:

- 초기 검증 단계
- 외부 연동 비용을 아끼고 싶은 경우

### 방식 B. 은행 또는 핀테크 API 연동

- 장점: 자동화 수준이 높음
- 단점: 연동 비용과 운영 복잡도가 있음

적합 상황:

- 주문량이 일정 수준 이상인 매장
- 반복 입금 확인 시간이 큰 매장

### 방식 C. 가상계좌 또는 고유 식별 금액 방식

- 장점: 매칭 정확도가 높음
- 단점: 고객 경험이 다소 달라질 수 있음

적합 상황:

- 오입금이 잦은 경우
- 같은 금액 주문이 자주 발생하는 경우

## 11. 추천 구현 순서

### 1단계

- 최종 금액 확정 상태 추가
- 주문별 기대 금액 저장
- 입금 상태 세분화
- 수동 확인 화면 개선

### 2단계

- 거래 내역 불러오기 기능
- 자동 대조 로직
- 검토 필요 큐

### 3단계

- 실시간 연동
- 가상계좌 또는 고유 식별 규칙 적용
- 자동 승인 정확도 고도화

## 12. 감사 로그와 보안

- 자동 확인 결과와 수동 확인 결과를 모두 기록
- 변경한 관리자 계정을 남김
- 거래 원본 데이터는 수정 불가 로그로 보관
- 주문 상태가 자동 변경될 때 이력 남김

## 13. 성공 판단 기준

- 입금 확인에 쓰는 평균 시간이 감소
- 검토 필요 건수 비율이 점진적으로 안정화
- 잘못된 자동 승인 사례가 거의 없음
- 입금 후 작업 시작까지의 평균 시간이 단축
