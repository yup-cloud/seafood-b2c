# 구현 청사진

## 1. 문서 목적

이 문서는 현재까지 정리된 아이디어 문서를 실제 개발 가능한 구조로 묶어주는 기준 문서다.  
핵심 목표는 아래 세 가지다.

- 기능을 구현 단위로 분해한다
- 어떤 기능을 먼저 만들어야 다음 단계가 가능한지 정리한다
- 기획, 화면, 데이터, 운영 규칙이 서로 어긋나지 않게 기준을 잡는다

## 2. 제품 구현의 큰 축

이 서비스는 아래 6개 모듈로 나누면 구현 순서가 선명해진다.

### M1. 고객 주문 접수/주문 가이드

- 오늘 시세/주문 가이드
- 품절/예약 가능 여부 안내
- 외부 웹 주문서
- 주문 완료 화면
- 주문 기본 정보 수집

### M2. 관리자 주문 운영

- 대시보드
- 주문 목록
- 주문 상세
- 상태 변경

### M3. 작업/출고 관리

- 손질 대기 목록
- 작업 큐
- 직접 픽업 / 퀵 / 택배 출고 처리
- 포장 완료 후 인계 관리

### M4. 정산/결제/입금 확인

- 최종 금액 확정
- 결제 누락 방지
- 수동 입금 확인
- 입금 상태 관리
- 검토 필요 주문 분리

### M5. 반절 주문 매칭

- 반절 구매 희망 접수
- 매칭 대기 관리
- 매칭 그룹 생성

### M6. 운영 자동화

- 자동 입금 대조
- 알림/안내 자동화
- 감사 로그 강화

## 3. 구현 단계 기준

### 1단계: 주문을 빠짐없이 받는다

이 단계에서는 고객이 오늘 주문 가능 품목과 규칙을 보고 주문서를 제출하고, 매장에서 주문을 놓치지 않고 볼 수 있어야 한다.

필수 포함:

- 오늘 시세/주문 가이드
- 고객 주문서
- 주문 생성
- 관리자 주문 목록
- 주문 상세

### 2단계: 주문을 현장에서 처리한다

이 단계에서는 최종 금액 확정, 입금 확인, 손질, 포장, 직접 픽업, 퀵, 택배까지 운영 흐름이 돌아가야 한다.

필수 포함:

- 최종 금액 확정
- 입금 상태 변경
- 작업 큐
- 주문 상태 전환
- 출고 관리 화면
- 모바일 관리자 사용성

### 3단계: 운영을 덜 헷갈리게 만든다

이 단계에서는 반절 매칭, 검토 큐, 메모, 필터처럼 현장 혼선을 줄이는 기능을 붙인다.

필수 포함:

- 반절 매칭 보드
- 입금 검토 큐
- 예외 처리 흐름

### 4단계: 운영을 자동화한다

이 단계에서는 반복 작업을 줄이는 자동 기능을 추가한다.

필수 포함:

- 자동 입금 대조
- 자동 상태 반영
- 고급 알림/후속 처리

## 4. 권장 구축 순서

기능은 아래 순서로 만드는 것이 가장 안전하다.

1. 주문 데이터 구조 확정
2. 오늘 시세/주문 가이드 구현
3. 고객 주문서 구현
4. 관리자 주문 목록과 상세 화면 구현
5. 최종 금액 확정과 수동 입금 확인 구현
6. 작업 큐 구현
7. 출고 관리 화면 구현
8. 모바일 관리자 보완
9. 반절 주문 옵션과 매칭 상태 추가
10. 반절 매칭 보드 구현
11. 거래 내역 기반 입금 검토 큐 구현
12. 자동 입금 대조 구현

## 5. 공통 역할 정의

### 고객

- 주문서를 제출한다
- 입금한다
- 픽업 또는 배송을 기다린다

### 사장님

- 주문 상태를 총괄한다
- 입금과 예외 상황을 최종 확인한다
- 매칭 실패나 환불 여부를 판단한다

### 직원

- 손질과 포장을 진행한다
- 작업 메모를 확인한다
- 작업 상태를 변경한다

## 6. 기준 상태 모델

상태값은 여러 문서에서 다르게 늘어나지 않도록 기준을 먼저 정해야 한다.

### 6-1. 기본 주문 상태

- `new`: 신규 접수
- `pricing_pending`: 최종 금액 확정 전
- `waiting_payment`: 입금 대기
- `payment_review`: 입금 검토 필요
- `ready_for_prep`: 손질 대기
- `in_prep`: 손질 중
- `packed`: 포장 완료
- `ready_for_handoff`: 인계/출고 대기
- `completed`: 픽업/퀵/택배 완료
- `cancelled`: 취소

### 6-2. 반절 주문 전용 상태

- `match_not_needed`: 일반 주문
- `matching_waiting`: 매칭 대기
- `matching_review`: 후보 검토 중
- `matched`: 매칭 완료
- `match_failed`: 매칭 실패

### 6-3. 입금 상태

- `unpaid`: 미입금
- `auto_confirmed`: 자동 확인 완료
- `manual_confirmed`: 수동 확인 완료
- `review_required`: 검토 필요
- `partial_paid`: 부분 입금
- `over_paid`: 초과 입금

### 6-4. 수령/출고 상태

- `pickup_waiting`: 직접 픽업 대기
- `pickup_done`: 직접 픽업 완료
- `quick_waiting`: 퀵 접수 대기
- `quick_sent`: 퀵 기사 인계 완료
- `parcel_waiting`: 택배 발송 대기
- `parcel_sent`: 택배 발송 완료

## 7. 핵심 데이터 엔터티

구현 초기에 아래 엔터티를 기준으로 잡으면 이후 기능 확장이 쉽다.

### Order

- 주문의 최상위 엔터티
- 고객 정보, 수령 정보, 상태값, 메모를 가진다

### OrderItem

- 품목, 규격, 수량, 손질 옵션을 가진다

### PriceBoardItem

- 오늘 시세, 품절 여부, 예약 가능 여부를 가진다

### QuoteRecord

- 최종 금액, 손질비, 운임비, 확정 메모를 가진다

### PaymentRecord

- 기대 입금 금액과 실제 입금 연결 상태를 가진다

### FulfillmentRecord

- 수령 방식, 수령인 정보, 주소, 송장 또는 인계 정보를 가진다

### BankTransaction

- 실제 계좌 거래 데이터를 저장한다

### MatchGroup

- 반절 주문 2건을 묶는 그룹 엔터티다

### AuditLog

- 상태 변경, 입금 확인, 매칭 처리 기록을 남긴다

## 9. 단계별 완료 기준

### 단계 1 완료 기준

- 고객이 오늘 주문 가능 품목과 규칙을 볼 수 있다
- 고객 주문이 폼 제출 후 누락 없이 저장된다
- 관리자 목록에서 오늘 주문을 확인할 수 있다
- 주문 상세에서 기본 정보가 읽힌다

### 단계 2 완료 기준

- 최종 금액을 확정한 뒤 입금 요청을 할 수 있다
- 주문 상태를 운영 흐름에 맞춰 변경할 수 있다
- 입금 여부를 수동으로 관리할 수 있다
- 작업 큐로 손질 순서를 확인할 수 있다
- 직접 픽업 / 퀵 / 택배 출고 대기를 구분해서 볼 수 있다

### 단계 3 완료 기준

- 반절 주문을 일반 주문과 구분해서 관리할 수 있다
- 반절 주문 매칭을 수동 확정할 수 있다
- 입금 검토 필요 주문을 따로 볼 수 있다

### 단계 4 완료 기준

- 거래 내역이 자동 또는 반자동으로 들어온다
- 명확한 거래는 자동 확인된다
- 애매한 거래는 검토 큐로 분리된다

## 10. 개발 중 자주 생길 리스크

### 상태값이 화면마다 다르게 정의되는 문제

- 해결: 상태 사전 문서를 기준으로 화면과 API를 통일

### 반절 매칭과 일반 주문 흐름이 섞이는 문제

- 해결: 주문 상태와 매칭 상태를 분리

### 자동 입금확인이 너무 일찍 들어가 복잡도가 커지는 문제

- 해결: 수동 확인 구조를 먼저 완성한 뒤 자동 대조를 붙임

### 최종 금액이 확정되기 전에 입금 로직을 태우는 문제

- 해결: `pricing_pending` 상태를 두고 정산 확정 이후에만 입금 흐름 진입

### 모바일 운영성이 뒤로 밀리는 문제

- 해결: 단계 2 완료 기준에 모바일 관리자 사용성을 포함

### 픽업/퀵/택배 흐름이 주문 상태와 뒤엉키는 문제

- 해결: 주문 상태와 수령/출고 상태를 분리
