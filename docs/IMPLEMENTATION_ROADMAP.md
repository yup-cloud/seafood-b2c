# 단계별 구현 로드맵

## 1. 로드맵 목적

이 문서는 기능을 한 번에 다 만들지 않고, 현장에 빨리 올릴 수 있는 순서대로 구현하기 위한 로드맵이다.

원칙은 아래와 같다.

- 먼저 주문 누락을 막는다
- 다음으로 운영 흐름을 안정화한다
- 그 다음 차별화 기능과 자동화를 붙인다

## 2. 전체 단계 요약

### Phase 0. 기준 정리

- 목표: 상태값, 데이터 구조, 화면 범위를 확정
- 결과물: 구현 기준 문서, 화면 우선순위, 엔터티 초안

### Phase 1. 핵심 MVP

- 목표: 주문 접수부터 관리자 확인까지 동작
- 결과물: 고객 주문서, 주문 완료 화면, 관리자 목록, 주문 상세, 수령 방식 분기

### Phase 2. 운영 MVP

- 목표: 입금 확인, 손질, 포장, 직접 픽업, 퀵, 택배 흐름이 돌아감
- 결과물: 작업 큐, 출고 관리 화면, 상태 전환, 필터, 모바일 관리자 보완

### Phase 3. 현장 고도화

- 목표: 반절 주문 운영과 결제 구조를 강화
- 결과물: 반절 매칭 보드, 결제 링크, 예약금/잔금 흐름, 검토 큐, 예외 처리 흐름

### Phase 4. 자동화

- 목표: 반복 확인 업무를 줄임
- 결과물: 자동 입금 대조, 자동 상태 반영, 알림 자동화, 고도화 로그

## 3. Phase 0. 기준 정리

### 목표

- 여러 문서에 흩어진 상태와 용어를 통일한다
- 구현 순서를 합의할 수 있게 만든다

### 해야 할 일

- 주문 상태 기준 확정
- 최종 금액 확정 상태 기준 확정
- 입금 상태 기준 확정
- 반절 매칭 상태 기준 확정
- 핵심 데이터 엔터티 정리
- 수령 방식/가공 방식 정책 매트릭스 정리
- 화면 우선순위 확정

### 완료 기준

- 개발자가 상태값 때문에 다시 기획 질문을 하지 않아도 됨
- 각 화면이 어떤 데이터를 쓰는지 설명 가능함

## 4. Phase 1. 핵심 MVP

### 목표

손님이 주문서를 제출하고, 매장이 관리자 화면에서 주문을 놓치지 않고 확인할 수 있게 한다.

### 포함 기능

- 오늘 시세/주문 가이드
- 고객 주문서
- 주문 완료 화면
- 주문 생성 API
- 관리자 주문 목록
- 주문 상세 기본 화면
- 검색
- 직접 픽업 / 퀵 / 택배 선택과 기본 수령 정보 저장
- 품목별 주문 가능 여부와 기본 마감 안내

### 제외 기능

- 자동 입금확인
- 반절 매칭 자동화
- 복잡한 통계

### 선행 조건

- 주문 데이터 구조 확정
- 기본 상태값 확정

### 완료 기준

- 주문서 제출 시 주문이 저장된다
- 관리자 화면에서 신규 주문을 확인할 수 있다
- 고객 연락처, 품목, 손질, 수령 시간이 누락 없이 보인다

## 5. Phase 2. 운영 MVP

### 목표

주문을 받는 수준을 넘어서 실제 매장 운영이 가능하도록 만든다.

### 포함 기능

- 최종 금액 확정
- 수동 입금 확인
- 주문 상태 전환
- 주문 마감 시간 노출
- 예상 금액 구조 안내
- 손질 대기 / 손질 중 / 포장 완료 흐름
- 작업 큐
- 직접 픽업 / 퀵 / 택배 출고 상태 관리
- 출고 관리 화면
- 오늘 주문 / 미입금 / 손질 대기 필터
- 모바일 관리자 사용성 개선

### 선행 조건

- Phase 1 완료

### 완료 기준

- 매장이 최종 금액을 확정하고 입금 요청을 보낼 수 있다
- 사장님이 계좌 확인 후 입금 상태를 바꿀 수 있다
- 고객이 홈과 주문서에서 오늘 마감 시간과 가격 구조를 이해할 수 있다
- 직원이 작업 큐에서 우선순위를 보고 상태를 바꿀 수 있다
- 고객 문의 시 주문 상세에서 현재 상태를 바로 확인할 수 있다
- 직접 픽업 / 퀵 / 택배 주문을 구분해서 출고 처리할 수 있다

## 6. Phase 3. 현장 고도화

### 목표

일반 주문 운영을 해치지 않으면서 반절 주문, 결제 링크, 예약금 흐름을 관리할 수 있게 한다.

### 포함 기능

- 주문서의 반절 구매 옵션
- 매칭 대기 상태
- 반절 매칭 관리 화면
- 관리자 수동 매칭
- 매칭 실패 처리
- 일반 주문 결제 링크 생성
- 예약 주문 예약금 결제
- 예약 주문 잔금 관리
- 입금 검토 필요 큐

### 선행 조건

- Phase 2 완료
- 반절 주문 운영 규칙 확정

### 완료 기준

- 반절 주문을 일반 주문과 섞지 않고 관리할 수 있다
- 관리자가 후보 주문을 보고 두 주문을 연결할 수 있다
- 고객이 링크를 통해 결제를 완료할 수 있다
- 예약 주문은 예약금 확인 후에만 다음 단계로 이동한다
- 애매한 입금 건을 검토 큐에서 별도로 처리할 수 있다

## 7. Phase 4. 자동화

### 목표

사장님이 반복해서 확인하던 업무를 줄인다.

### 포함 기능

- 거래 내역 불러오기
- 자동 입금 대조
- 명확한 거래 자동 승인
- 애매한 거래 자동 분류
- 단계별 알림 발송
- 자동 이력 기록

### 선행 조건

- Phase 3 완료
- 거래 데이터 확보 방식 결정

### 완료 기준

- 반복적인 수동 입금 확인 건수가 감소한다
- 자동 승인 오탐이 낮게 유지된다
- 검토가 필요한 거래만 따로 볼 수 있다
- 고객이 결제/준비/출고 상태를 문의 없이 확인할 수 있다

## 8. Phase별 화면 우선순위

### Phase 1

- 고객 주문서
- 주문 완료 화면
- 관리자 주문 목록
- 주문 상세

### Phase 2

- 작업 큐
- 출고 관리 화면
- 모바일 주문 상세
- 대시보드 필터 강화

### Phase 3

- 반절 매칭 관리 화면
- 입금 검토 큐

### Phase 4

- 거래 내역 모니터링 화면
- 자동 매칭 로그 화면

## 9. 권장 스프린트 분할 예시

### Sprint 1

- 데이터 구조
- 오늘 시세/주문 가이드 구조
- 고객 주문서
- 주문 생성
- 수령 방식 분기

### Sprint 2

- 관리자 목록
- 주문 상세
- 검색/필터 기본형

### Sprint 3

- 최종 금액 확정
- 수동 입금 확인
- 주문 마감 시간 노출
- 예상 금액 구조 노출
- 작업 큐
- 상태 전환
- 출고 상태 기본형

### Sprint 4

- 모바일 관리자 보완
- 출고 관리 화면
- 운영 예외 메모
- QA

### Sprint 5

- 반절 매칭 옵션
- 매칭 대기 상태
- 매칭 보드 기본형
- 결제 링크 구조
- 예약금/잔금 상태 설계

### Sprint 6

- 거래 내역 연동 또는 업로드
- 입금 검토 큐
- 자동 대조 기본형
- 상태 알림 자동화

## 10. 구현 순서상 의존성

### 반절 매칭은 무엇에 의존하는가

- 주문 생성 구조
- 주문 상태 체계
- 수령 방식 정책
- 입금 확정 흐름

### 자동 입금확인은 무엇에 의존하는가

- 최종 금액 확정 구조
- 주문별 기대 입금 금액 저장
- 수동 입금 확인 UI
- 감사 로그 구조

## 11. 배포 전략 제안

### 1차 배포

- Phase 1 + Phase 2 핵심 기능만 배포
- 실제 매장에서 운영 테스트

### 2차 배포

- 반절 매칭 기능 배포
- 운영 규칙 보정

### 3차 배포

- 자동 입금확인 기능 배포
- 자동 승인 기준 보수적으로 적용

## 12. 매 단계에서 꼭 확인할 것

- 고객이 헷갈리는 입력 항목은 없는가
- 사장님이 3초 안에 상태를 이해할 수 있는가
- 직원이 휴대폰으로 상태를 바꾸기 쉬운가
- 예외 상황이 생겼을 때 사람이 개입할 수 있는가
- 직접 픽업 / 퀵 / 택배 주문이 섞여도 출고가 꼬이지 않는가
