# 제품 백로그

## 1. 문서 목적

이 문서는 기능을 우선순위와 구현 단위로 쪼개기 위한 백로그다.  
우선순위 기준은 아래와 같다.

- `P0`: 없으면 MVP가 성립하지 않음
- `P1`: MVP 이후 빠르게 붙여야 효과가 큼
- `P2`: 이후 고도화 단계

## 2. Epic A. 고객 주문 접수

### A-01. 고객은 주문서를 제출할 수 있어야 한다

- 우선순위: `P0`
- 설명: 고객이 카톡 링크를 통해 주문 정보를 입력한다
- 수용 기준:
  - 주문자명, 연락처, 품목, 수량, 손질, 수령 시간이 저장된다
  - 필수 항목 누락 시 제출되지 않는다
  - 모바일 브라우저에서 제출 가능하다

### A-05. 고객은 오늘 주문 가능 품목과 마감 규칙을 볼 수 있어야 한다

- 우선순위: `P0`
- 설명: 오픈카톡 공지를 대체하는 주문 가이드가 필요하다
- 수용 기준:
  - 오늘 시세 또는 주문 가능 품목이 보인다
  - 품절/완판 상태를 구분해서 볼 수 있다
  - 예약 주문 가능 여부와 마감 시간이 보인다

### A-06. 고객은 수령 방식에 맞지 않는 가공 조합에 대해 안내받아야 한다

- 우선순위: `P0`
- 설명: 일반택배 + 회처럼 위험한 주문 조합을 줄여야 한다
- 수용 기준:
  - 수령 방식에 따라 경고 또는 제한이 노출된다
  - 회/오로시/원물 선택에 대한 안내가 분기된다

### A-04. 고객은 수령 방식을 선택할 수 있어야 한다

- 우선순위: `P0`
- 설명: 직접 픽업, 퀵, 택배 중 하나를 선택한다
- 수용 기준:
  - 주문서에 수령 방식 선택이 있다
  - 퀵/택배 선택 시 수령인 정보와 주소 입력이 가능하다
  - 직접 픽업 선택 시 주소 입력이 강제되지 않는다

### A-02. 고객은 주문 접수 완료를 확인할 수 있어야 한다

- 우선순위: `P0`
- 설명: 주문이 정상 접수되었는지 알 수 있어야 한다
- 수용 기준:
  - 완료 화면에 주문 접수 메시지가 나온다
  - 입금 또는 후속 절차가 안내된다

### A-03. 고객은 반절 구매를 선택할 수 있어야 한다

- 우선순위: `P1`
- 설명: 전체 구매가 아닌 반절 구매 희망을 선택한다
- 수용 기준:
  - 주문서에 구매 단위 선택이 있다
  - 반절 선택 시 매칭 대기 안내가 노출된다

## 3. Epic B. 관리자 주문 운영

### B-01. 관리자는 신규 주문 목록을 볼 수 있어야 한다

- 우선순위: `P0`
- 설명: 오늘 들어온 주문을 확인한다
- 수용 기준:
  - 신규 주문이 목록에 표시된다
  - 기본 정보가 한 줄에서 보인다

### B-02. 관리자는 주문 상세를 확인할 수 있어야 한다

- 우선순위: `P0`
- 설명: 한 건의 주문을 끝까지 처리할 수 있어야 한다
- 수용 기준:
  - 고객 정보, 주문 품목, 수령 시간, 메모가 보인다
  - 상태 변경 이력이 보인다

### B-03. 관리자는 주문을 검색/필터할 수 있어야 한다

- 우선순위: `P0`
- 설명: 미입금, 손질 대기 등 필요한 주문만 본다
- 수용 기준:
  - 오늘 주문 필터가 있다
  - 미입금 필터가 있다
  - 이름 또는 주문번호 검색이 된다

## 4. Epic C. 입금 확인

### C-01. 관리자는 입금을 수동 확인할 수 있어야 한다

- 우선순위: `P0`
- 설명: 입금 여부를 체크해서 작업으로 넘긴다
- 수용 기준:
  - 미입금에서 확인 완료로 바꿀 수 있다
  - 확인 시간과 담당자가 기록된다

### C-02. 관리자는 검토 필요한 입금을 구분할 수 있어야 한다

- 우선순위: `P1`
- 설명: 애매한 입금 건을 따로 모아 처리한다
- 수용 기준:
  - `검토 필요` 상태가 존재한다
  - 검토 큐에서 관련 정보가 함께 보인다

### C-03. 시스템은 입금을 자동 대조할 수 있어야 한다

- 우선순위: `P1`
- 설명: 실제 거래와 주문을 자동 비교한다
- 수용 기준:
  - 명확한 거래는 자동 확인 후보가 된다
  - 애매한 거래는 자동 승인되지 않는다

### C-04. 관리자는 최종 금액을 확정할 수 있어야 한다

- 우선순위: `P0`
- 설명: 원물 가격, 손질비, 운임비를 합산해 결제 기준 금액을 만든다
- 수용 기준:
  - 최종 금액 확정 전 상태가 존재한다
  - 최종 금액과 세부 비용을 저장할 수 있다
  - 확정 후 입금 요청 상태로 전환할 수 있다

### C-05. 미입금 주문은 출고 전에 다시 확인되어야 한다

- 우선순위: `P0`
- 설명: 결제 누락 출고를 막아야 한다
- 수용 기준:
  - 미입금 상태에서 출고 완료 처리 시 경고가 뜬다
  - 강제 진행 여부가 기록된다

### C-06. 고객은 결제 링크로 쉽게 결제할 수 있어야 한다

- 우선순위: `P1`
- 설명: 계좌 확인 부담을 줄이려면 결제 자체를 구조 안에 넣어야 한다
- 수용 기준:
  - 최종 금액 확정 후 결제 링크를 생성할 수 있다
  - 결제 완료 시 주문 상태가 자동 반영된다
  - 실패 또는 미결제 상태를 다시 확인할 수 있다

### C-07. 예약 주문은 예약금 결제 후 진행되어야 한다

- 우선순위: `P1`
- 설명: 예약 주문은 일반 주문과 다른 결제 흐름이 필요하다
- 수용 기준:
  - 예약 주문에 예약금 금액을 저장할 수 있다
  - 예약금 결제 완료 전에는 확보 진행으로 넘기지 않는다
  - 예약금 확인 후 다음 단계로 자동 전환된다

### C-08. 예약 주문은 잔금 결제 상태를 따로 관리할 수 있어야 한다

- 우선순위: `P1`
- 설명: 예약금과 잔금이 섞이면 운영과 고객 안내가 모두 헷갈린다
- 수용 기준:
  - 예약금과 잔금 상태가 분리되어 보인다
  - 잔금 결제 링크를 따로 보낼 수 있다
  - 출고 전 잔금 미결제가 경고된다

## 5. Epic D. 작업 관리

### D-01. 직원은 손질 대기 주문을 볼 수 있어야 한다

- 우선순위: `P0`
- 설명: 어떤 주문부터 작업할지 알아야 한다
- 수용 기준:
  - 수령 시간 기준 정렬이 된다
  - 손질 방식과 요청사항이 보인다

### D-02. 직원은 작업 상태를 변경할 수 있어야 한다

- 우선순위: `P0`
- 설명: 손질 중, 포장 완료 등의 상태를 바꾼다
- 수용 기준:
  - 작업 큐나 상세 화면에서 상태 변경 가능
  - 변경 이력이 저장된다

### D-03. 관리자는 모바일에서 작업 상태를 바꿀 수 있어야 한다

- 우선순위: `P0`
- 설명: 매장 안에서 휴대폰으로 운영한다
- 수용 기준:
  - 모바일 화면에서 주요 버튼이 누르기 쉽다
  - 상세 정보가 잘리지 않는다

### D-04. 관리자는 출고 대기 주문을 방식별로 볼 수 있어야 한다

- 우선순위: `P0`
- 설명: 직접 픽업, 퀵, 택배 주문을 나눠서 관리한다
- 수용 기준:
  - 직접 픽업 대기 목록이 보인다
  - 퀵 출고 대기 목록이 보인다
  - 택배 발송 대기 목록이 보인다

### D-05. 관리자는 퀵/택배 출고 정보를 기록할 수 있어야 한다

- 우선순위: `P1`
- 설명: 퀵 인계와 택배 송장 정보를 남긴다
- 수용 기준:
  - 퀵 기사 인계 메모를 저장할 수 있다
  - 택배 송장번호를 저장할 수 있다
  - 출고 완료 여부가 기록된다

### D-06. 관리자는 일반택배/당일택배/고속택배 정책을 관리할 수 있어야 한다

- 우선순위: `P1`
- 설명: 택배도 세부 운영 규칙이 다르다
- 수용 기준:
  - 세부 택배 유형을 구분할 수 있다
  - 마감 시간과 제한 메모를 보여줄 수 있다

## 6. Epic E. 반절 주문 매칭

### E-01. 반절 주문은 매칭 대기 상태로 관리되어야 한다

- 우선순위: `P1`
- 설명: 일반 주문과 구분해서 관리한다
- 수용 기준:
  - 반절 주문에 매칭 상태가 저장된다
  - 목록에서 매칭 대기 주문을 볼 수 있다

### E-02. 관리자는 반절 주문 후보를 연결할 수 있어야 한다

- 우선순위: `P1`
- 설명: 두 주문을 하나의 매칭 그룹으로 묶는다
- 수용 기준:
  - 매칭 화면에서 후보 주문을 본다
  - 수동으로 두 주문을 연결한다
  - 연결 이력이 남는다

### E-03. 시스템은 유사 반절 주문을 추천할 수 있어야 한다

- 우선순위: `P2`
- 설명: 품목, 규격, 시간 기준으로 후보를 추천한다
- 수용 기준:
  - 추천 후보가 점수와 함께 표시된다
  - 자동 확정은 하지 않는다

### F-04. 예약 주문은 오늘 주문과 분리해서 관리되어야 한다

- 우선순위: `P1`
- 설명: 내일 물건 확보 요청이나 마감 시간 있는 주문이 있다
- 수용 기준:
  - 예약 주문 날짜를 저장할 수 있다
  - 오늘 주문 목록과 분리해서 볼 수 있다
  - 경매 확보 요청 메모를 남길 수 있다

## 10. Epic I. 고객 신뢰와 재구매

### I-01. 고객은 주문 마감 시간을 쉽게 볼 수 있어야 한다

- 우선순위: `P0`
- 설명: 오늘 주문 가능한지 먼저 보여줘야 문의가 줄어든다
- 수용 기준:
  - 홈과 주문서에 수령 방식별 마감 시간이 보인다
  - 예약 주문 가능 여부가 함께 보인다

### I-02. 고객은 예상 금액 구조를 주문 전에 이해할 수 있어야 한다

- 우선순위: `P0`
- 설명: 최종 금액이 나중에 정해져도 가격 구조는 미리 이해해야 한다
- 수용 기준:
  - 시세 기반 예상 범위가 노출된다
  - 손질비와 운임비가 별도 항목으로 안내된다
  - 예약 주문일 경우 예약금 정책이 함께 보인다

### I-03. 고객은 토큰 입력 없이 안내 링크로 주문 상태를 확인할 수 있어야 한다

- 우선순위: `P1`
- 설명: 상태 조회 진입이 번거로우면 고객은 다시 문의하게 된다
- 수용 기준:
  - 발송 링크만 눌러도 상태 조회 페이지가 열린다
  - 토큰 입력은 보조 수단으로 남는다

### I-04. 고객은 지난 주문을 바탕으로 다시 주문할 수 있어야 한다

- 우선순위: `P2`
- 설명: 재주문 경험이 좋아야 단골 전환이 빨라진다
- 수용 기준:
  - 최근 주문 내역을 다시 불러올 수 있다
  - 자주 쓰는 손질/포장 조합을 빠르게 적용할 수 있다
