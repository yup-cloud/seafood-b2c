# MVP 기능명세서

## 1. 제품 한 줄 정의

오픈카톡으로 들어오는 수산시장 소매 주문을 `오늘 시세/주문 가이드 + 외부 웹 주문서`로 표준화하고, 최종 금액 확정, 입금 확인, 손질, 직접 픽업, 퀵, 택배 출고까지 한 화면에서 처리하는 소매점용 운영 SaaS.

## 2. 핵심 사용자

- 사장님: 주문 누락 없이 접수 현황과 입금 상태를 빠르게 확인해야 함
- 직원: 손질 순서, 픽업 시간, 특이 요청을 헷갈리지 않고 처리해야 함
- 고객: 카톡으로 길게 설명하지 않고 주문 내용을 한 번에 정확하게 넣고 싶어 함

## 3. 현재 운영 문제

### 3-1. 주문 접수 문제

- 오픈카톡 대화방에서 주문 정보가 메시지 단위로 흩어짐
- 고객마다 주문 형식이 달라서 다시 물어보는 일이 많음
- 품목, 수량, 손질 방식, 픽업 시간 중 하나라도 빠지면 현장 혼선이 생김

### 3-2. 입금 확인 문제

- 입금자명과 주문자명이 달라 대조 시간이 오래 걸림
- 선입금 확인이 안 된 주문이 손질로 넘어가는 경우가 생김
- 주문별 입금 확인 여부를 직원끼리 공유하기 어려움

### 3-3. 작업 관리 문제

- 어떤 주문을 먼저 손질해야 하는지 우선순위가 눈에 안 들어옴
- 픽업 예정 시간과 손질 완료 시간을 따로 적거나 기억해야 함
- 고객 요청사항이 카톡 메시지 안에 묻혀 누락되기 쉬움

### 3-4. 인수인계 문제

- 사장님, 직원, 가족이 번갈아 주문을 받으면 상태가 서로 다르게 이해됨
- 전화 문의가 오면 현재 주문 상태를 바로 답하기 어려움
- 하루 마감 후 어떤 주문이 완료됐고 어떤 주문이 남았는지 정리하기 번거로움

### 3-5. 수령/출고 문제

- 손님마다 직접 픽업, 퀵, 택배 방식이 달라 필요한 정보가 달라짐
- 퀵은 주소와 연락처, 택배는 수령인 정보와 포장 상태가 빠지면 바로 출고가 어려움
- 포장 완료 후에도 어떤 주문이 픽업 대기인지, 어떤 주문이 퀵/택배 출고 대기인지 헷갈리기 쉬움
- 택배 발송 후 송장번호, 퀵 출발 후 인계 여부를 따로 관리하지 않으면 문의 대응이 어려워짐

### 3-6. 판매 단위 문제

- 손님은 원물 전체가 아니라 반절만 원하는데, 현재는 맞는 상대를 찾기 어려워 주문을 놓치는 경우가 있음
- 매장은 원물 한 마리를 기준으로 판매하고 싶지만, 소량 수요를 받는 순간 계산과 예약 관리가 복잡해짐
- 반절 주문을 임의로 받으면 다른 반절이 안 붙었을 때 취소나 응대 문제가 생길 수 있음

### 3-7. 시세/품절 공지 문제

- 고객은 시세표에 없는 품목이 정말 없는지 반복 문의함
- 시세 업데이트 시간, 완판 여부, 예약 가능 여부를 계속 다시 물어봄
- 오늘 주문 가능한 품목과 이미 끝난 품목을 구분해서 보여줄 필요가 큼

### 3-8. 정산/결제 문제

- 주문을 받았는데 최종 금액이 아직 확정되지 않은 경우가 생김
- 입금 전에 출고가 되거나, 영수증/정산 처리가 뒤로 밀릴 위험이 있음
- 손질비, 택배비, 퀵비가 붙는 주문은 최종 금액 확인 흐름이 필요함

### 3-9. 예약/사전주문 문제

- 다음날 경매 물건을 잡아달라는 예약 요청이 존재함
- 품목별로 주문 마감 시간이 다를 수 있음
- 즉시 주문과 예약 주문을 같은 흐름으로 처리하면 누락되기 쉬움

## 4. MVP 설계 원칙

- 오픈카톡은 유입 채널로 유지하고, 주문 확정은 외부 주문서에서 받음
- 복잡한 커머스 플랫폼이 아니라 현장 운영 흐름을 줄이는 도구로 설계함
- 첫 버전은 주문 정확도와 작업 속도 개선에 집중함
- 결제 자동화, 재고 연동, 배달 최적화 같은 확장 기능은 제외함
- 다만 반절 매칭, 자동 입금확인처럼 현장 효율이 큰 기능은 확장 우선 후보로 설계 여지를 남김

## 5. MVP 범위

### 포함

- 고객용 오늘 시세/주문 가이드
- 고객용 외부 웹 주문서
- 주문 접수 후 관리자 화면 자동 생성
- 최종 금액 확정 및 입금 요청
- 주문 상태 관리
- 입금 확인 체크
- 손질 옵션/비용 관리
- 손질 요청 및 픽업 시간 관리
- 직접 픽업 / 퀵 / 택배 수령 방식 관리
- 주문 메모와 고객 요청사항 관리
- 예약 주문 관리
- 오늘 주문 중심의 작업 큐

### 제외

- 카드결제, 간편결제, PG 연동
- 재고 자동 차감
- 회계, 세금계산서, POS 연동
- 고객용 회원가입/앱
- 기사 배차 및 배송 경로 최적화
- 매장 여러 지점 통합 관리
- 오픈카톡 자체 커뮤니티 기능 대체

## 6. 필수 기능

### 6-1. 고객 주문서

- 오늘 주문 가능 품목 또는 예약 주문 가능 여부 안내
- 주문자명, 연락처, 입금자명 입력
- 품목명/규격/수량 입력
- 손질 방식 선택
- 손질 비용과 추가 비용 안내
- 수령 방식 선택
  - 직접 픽업
  - 퀵
  - 택배
- 희망 수령 시간 입력
- 퀵 또는 택배일 경우 수령인명, 연락처, 주소 입력
- 수령 방식에 따라 허용 가능한 가공 방식 안내
- 요청사항 입력
- 주문 접수 후 확인 문구와 다음 단계 안내 제공

### 6-2. 오늘 시세/주문 가이드

- 오늘 주문 가능한 품목 표시
- 품절/완판 상태 표시
- 예약 가능 품목과 마감 시간 안내
- 직접 픽업 / 퀵 / 택배별 주문 조건 안내
- 회/오로시/원물별 권장 수령 방식 안내
- 영업시간, 주소, 계좌, 기본 주문 가이드 노출

### 6-3. 주문 관리

- 신규 주문 자동 등록
- 주문번호 자동 생성
- 상태값 관리
- 상태 기본안:
  - 최종 금액 확정 전
  - 입금 확인 전
  - 손질 대기
  - 손질 중
  - 포장 완료
  - 출고 대기
  - 완료
  - 취소

### 6-4. 입금 확인

- 입금 확인 여부 체크박스 또는 버튼
- 입금 확인 시 담당자와 시간 기록
- 입금 전 주문만 따로 모아보기
- 입금자명 검색 지원
- 자동화 확장 시 `자동 확인 완료`, `검토 필요`, `미입금` 상태를 지원할 수 있는 구조로 설계

### 6-5. 정산/최종 금액 확정

- 주문 접수 후 `최종 금액 확정 전` 상태를 둘 수 있어야 함
- 원물 가격, 손질비, 운임비를 합친 최종 금액 저장
- 고객에게 최종 금액과 입금 안내를 다시 전달
- 입금 전에는 출고 완료 처리 시 강한 경고 또는 차단
- 현금영수증/결제수단 메모 가능

### 6-6. 손질/작업 관리

- 손질 필요 여부 표시
- 손질 방식 요약 표시
- 희망 수령 시간 순 정렬
- 오늘 처리해야 할 작업 큐 제공
- 긴급 주문 또는 특이 요청 메모 표시

### 6-7. 수령/출고 관리

- 직접 픽업, 퀵, 택배별 필수 정보 관리
- 포장 완료 후 `픽업 대기`, `퀵 접수 대기`, `택배 발송 대기` 상태를 구분
- 퀵 기사 전달 메모 입력
- 택배 송장번호 기록
- 택배 세부 유형과 마감 규칙 관리
  - 일반택배
  - 당일택배
  - 고속택배
- 택배/가공 방식 조합 경고
- 수령 방식별 완료 처리
  - 직접 픽업: 고객 인수 완료
  - 퀵: 기사 인계 완료
  - 택배: 발송 완료 또는 배송 완료

### 6-8. 주문 상세

- 고객 정보, 주문 내용, 입금 정보, 작업 메모를 한 화면에 표시
- 최종 금액과 세부 비용 표시
- 수령 방식과 수령인 정보 표시
- 내부 메모 추가
- 상태 변경 이력 확인
- 고객 문의 대응용 핵심 정보 상단 고정

### 6-9. 관리자 사용성

- 모바일에서도 상태 변경 가능
- 오늘 주문 / 미입금 / 손질 대기 / 픽업 대기 / 퀵 대기 / 택배 대기 필터 제공
- 이름, 연락처, 주문번호 검색 가능

### 6-10. 예약 주문 관리

- 오늘 주문과 예약 주문을 구분해서 표시
- 희망 날짜 기준으로 주문 보관
- 경매 확보 요청 메모 저장
- 품목별 사전 주문 마감 시간 관리

## 7. 운영 흐름

1. 고객이 오픈카톡 공지 또는 링크에서 오늘 시세/주문 가이드를 확인함
2. 고객이 주문서에 품목, 수량, 손질, 수령 방식, 수령 시간을 입력함
3. 주문이 관리자 화면에 `주문 접수` 또는 `최종 금액 확정 전` 상태로 생성됨
4. 매장은 필요 시 최종 금액, 손질비, 운임비를 확정함
5. 고객이 입금하면 주문이 `입금 확인 전` 또는 `입금 완료` 상태로 이동함
6. 매장은 입금 확인 후 `손질 대기`로 변경함
7. 직원은 작업 큐에서 시간 순으로 손질을 진행함
8. 손질 완료 후 `포장 완료`로 변경함
9. 수령 방식에 따라 후속 처리함
   - 직접 픽업: `픽업 대기` 후 방문 시 완료
   - 퀵: 주소 확인 후 기사 인계 처리
   - 택배: 송장 등록 후 발송 처리

## 10. MVP 완료 기준

- 고객이 카톡 대화 없이 주문서를 스스로 제출할 수 있음
- 고객이 오늘 주문 가능 품목과 수령 규칙을 보고 스스로 판단할 수 있음
- 사장님이 주문 누락 없이 오늘 주문을 한 화면에서 확인할 수 있음
- 직원이 입금 여부와 손질 우선순위를 헷갈리지 않음
- 픽업, 퀵, 택배 주문 상태를 전화 없이 바로 확인할 수 있음
- 미입금 주문이 실수로 출고되지 않음

## 11. 확장 우선 후보

### 11-1. 반절 주문 매칭 기능

#### 기능 개념

- 고객이 주문서에서 `원물 전체` 또는 `반절 구매 희망`을 선택할 수 있음
- 반절 구매 희망 주문은 바로 확정되지 않고 `매칭 대기` 상태로 들어감
- 같은 품목, 비슷한 규격, 같은 수령 시간대의 주문끼리 묶어서 한 원물로 연결함
- 매칭이 완료되면 두 주문을 연결하고 최종 확정 안내 및 입금 요청을 보냄

#### 왜 중요한가

- 원물 전체가 부담스러운 손님도 받을 수 있어 잠재 수요가 늘어남
- 매장은 반절 단위 수요를 체계적으로 모아 한 마리 판매 기회를 높일 수 있음
- 단골이 아닌 1~2인 가구, 소규모 모임 수요까지 흡수할 수 있음

#### 운영상 필요한 규칙

- 매칭 대기 허용 시간
- 같은 품목/규격 기준
- 매칭 실패 시 처리 방식
  - 취소
  - 전체 구매 제안
  - 다음 입고 일정 안내
- 반절별 손질 방식이 달라도 가능한 범위 정의

#### 추천 도입 방식

- 1단계: 관리자 수동 매칭 보드
- 2단계: 조건 기반 자동 추천
- 3단계: 고객에게 실시간 매칭 진행 상태 안내

### 11-2. 자동 입금확인 기능

#### 기능 개념

- 주문별 기대 입금 금액과 입금자명을 저장해두고 실제 입금 내역과 자동 대조
- 일치 시 주문 상태를 자동으로 `입금 확인 완료`로 변경
- 애매한 건은 `검토 필요`로 남겨 사람이 마지막 확인만 하게 함

#### 구현 방식 후보

- 은행 또는 핀테크 입출금 내역 연동
- 가상계좌 방식
- 주문별 고유 입금금액 또는 식별 코드를 활용한 자동 매칭

#### 왜 중요한가

- 사장님이 계좌 앱과 주문 목록을 번갈아 볼 필요가 줄어듦
- 주문자명과 입금자명이 달라도 자동 대조 정확도를 높일 수 있음
- 미입금 주문과 입금 완료 주문이 즉시 구분되어 작업 속도가 빨라짐

#### 운영상 필요한 규칙

- 자동 일치 기준
  - 금액
  - 입금자명
  - 시간 범위
- 부분 일치 시 처리 방식
- 중복 입금 또는 오입금 예외 처리

#### 권장 우선순위

- 자동 입금확인은 운영 효율 효과가 커서 MVP 직후 우선 개발 후보로 둠
- 첫 단계는 자동 매칭 + 사람 검토 조합으로 시작하는 것이 안전함
